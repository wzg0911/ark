### 关键设计决策

**1. Zero-overhead 关闭路径（默认行为）**
```python
def _emit_otel(event_type, tool_name, **attrs):
    if not os.getenv("ARK_OTEL_ENDPOINT", ""):  # O(1) 一次if
        return
    # ... OTel 真正逻辑
```
- `os.getenv` 是 C 实现，单次 < 1μs
- 99.99% 用户的零配置路径 = 单次 if 判断 + 单次 env 读取

**2. 函数内读取 env（陷阱修复）**
最初版在模块顶部用 `_OTEL_ENDPOINT = os.getenv(...)`，导致 `import ark` 时锁定值。改为函数内读取后，`os.environ["ARK_OTEL_ENDPOINT"] = "..."` 任何时候都生效 — 关键 for pytest 跨测试激活/禁用。

**3. 异常吞噬（可靠性铁律）**
```python
except Exception:
    pass  # 可靠性组件自身不能成为故障源
```
OTel exporter 失败 / 网络断开 / Collector 不存在 — 永远不抛，绝不污染主业务流。

**4. 真实事件负载**
- `intercepts` 事件携带 `saved_ms`（被拦截调用节省的真实耗时）
- `circuit.open` 事件携带 `failure_count` + `error` 摘要（前 200 字符）
- `validation.fail` 事件携带 `error_count` + `first_error` 摘要

### 8 种事件类型（全部自动触发）
| 事件 | 触发条件 | 携带字段 |
|------|----------|----------|
| `ark.idempotency.miss` | 工具首次调用 | tool_name, idempotency_key |
| `ark.guardian.intercept` | 幂等拦截（重复调用） | tool_name, idempotency_key, saved_ms, reason |
| `ark.circuit.open` | 熔断器打开 | tool_name, failure_count, threshold, error |
| `ark.circuit.half_open` | 半开探测 | tool_name, recovery_timeout |
| `ark.circuit.close` | 熔断器恢复 | tool_name, recovered |
| `ark.validation.pass` | 输出验证通过 | tool_name, schema |
| `ark.validation.fail` | 输出验证失败 | tool_name, schema, error_count, first_error |
| `ark.validation.fail` (null) | Agent 返回 None | tool_name, schema, reason=null_output |
